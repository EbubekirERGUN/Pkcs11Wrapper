# Vendor-native HSM audit integration evaluation (starting with Thales Luna)

See also:

- `docs/telemetry-redaction.md` for the current wrapper telemetry redaction contract
- `docs/telemetry-integrations.md` for the current `ILogger` / `ActivitySource` telemetry projection model
- `docs/luna-integration.md` for the practical Luna standard-PKCS#11 setup path
- `docs/luna-compatibility-audit.md` for the current Luna standard-vs-extension compatibility boundary
- `docs/luna-vendor-extension-design.md` for the future Luna `CA_*` extension-layer boundary

## Purpose

This document evaluates how `Pkcs11Wrapper` should think about **vendor-native HSM audit logs** without confusing them with the repo's existing **wrapper-side PKCS#11 telemetry**.

It starts with **Thales Luna** because the repo already has Luna compatibility, integration, and extension-boundary notes.

It is intentionally conservative:

- it does **not** claim that wrapper telemetry is a substitute for HSM-native audit logging
- it does **not** propose adding vendor-native audit ingestion to the core `Pkcs11Wrapper` API surface right now
- it separates **standard wrapper observability** from **vendor audit collection**
- it identifies realistic future directions by ingestion path: **CLI / syslog / export / API**
- it scopes future work narrowly enough that follow-up issues can be created without overcommitting implementation

## Current observability layers in this repo

The repo already has more than one observability concept. They should stay distinct.

| Layer | Current repo support | Source of truth | Scope | Strengths | Limits |
| --- | --- | --- | --- | --- | --- |
| Wrapper PKCS#11 telemetry | Yes | `IPkcs11OperationTelemetryListener` events emitted by `Pkcs11Wrapper` | Only operations executed through the current process using this wrapper instance | Fast, redacted, generic, easy to send to logs/traces | Cannot see activity from other clients or HSM-local actions outside this process |
| Admin-panel audit chain | Yes | Local append-only JSONL audit entries in the admin host | Admin-panel user actions on the current host/app | Good for local operator accountability and tamper-evident admin history | Not vendor-native, not HSM-global, not a hardware-originated audit stream |
| Vendor-native HSM audit logs | Not implemented in this repo | HSM/appliance/cloud vendor audit subsystem | Potentially all relevant HSM activity visible to that vendor platform | Captures HSM-native/global events, vendor roles, device-side operations, and sometimes integrity-verifiable records | Vendor-specific transport, schema, privileges, retention, and tooling |

## Wrapper telemetry vs vendor-native audit: the actual difference

The most important conclusion from issue #41 is that these are **complementary layers**, not competing names for the same thing.

### Wrapper telemetry is process-local observability

Current wrapper telemetry is emitted by the wrapper after major PKCS#11 operations and is designed for:

- application diagnostics
- latency/failure correlation
- safe structured logging/tracing
- understanding how *this process* used the PKCS#11 module

Its shape is deliberately generic and conservative:

- operation name + native PKCS#11 function name
- elapsed duration
- success / returned-false / failure outcome
- raw return value when available
- optional slot / session / mechanism context
- redacted fields only

That is useful, but it does **not** prove what the HSM globally observed.

### Vendor-native audit is HSM/platform accountability

Vendor-native audit logs are generated by the vendor platform itself and are designed for:

- compliance review
- cross-client accountability
- security event review
- proving that the HSM/appliance/cloud service recorded a specific operation or management action
- reviewing events outside the current application process

Compared to wrapper telemetry, vendor audit logs may include:

- all relevant clients, not just the current process
- HSM role / actor identity as understood by the vendor platform
- HSM management operations outside PKCS#11 consumer code
- vendor-only operations (`CA_*`, appliance tasks, HA/cloning/admin events, cloud-service control events)
- integrity or authenticity features tied to the HSM/platform

### Why the distinction matters architecturally

Because the two layers have different origins and guarantees:

- wrapper telemetry belongs in the **vendor-neutral core/app observability path**
- vendor audit belongs in a **vendor-specific operational ingestion path**

Trying to collapse vendor-native audit into the existing wrapper telemetry interface would blur those guarantees and create false expectations.

## Thales Luna findings

This section summarizes what is realistically available from public Thales Luna documentation reviewed for this issue.

### What Luna audit provides that wrapper telemetry does not

Public Thales documentation describes Luna audit logging as an HSM/platform feature rather than a normal wrapper-side diagnostic stream.

Key properties called out in the public docs:

- audit entries originate from the **Luna HSM / appliance side**, not from the client wrapper process
- logs are controlled by a dedicated **audit role / audit user** rather than ordinary application users
- event categories are configurable, including login, management, key management, key usage, first-use-only modes, and audit-management actions
- public Luna event catalogs include both standard `C_*` operations and many Luna-only `CA_*` operations
- logs can be rotated, exported, and verified
- remote audit logging can be sent to one or more remote log servers using `udp`, `tcp`, or `relp`, with newer firmware/appliance support for `tls`
- log authenticity/integrity is part of the vendor story through audit-log secrets and verification flows
- audit logging consumes HSM resources and can affect performance when configured aggressively
- if logging is mismanaged and logs fill up, the HSM can stop servicing normal operations until audit maintenance occurs

Those capabilities are materially different from the current wrapper telemetry model.

### What Luna audit appears to cover

Based on the reviewed public documentation, Luna audit can record far more than a generic wrapper listener can know on its own.

Examples explicitly shown in public Luna docs include:

- standard PKCS#11 access/session events such as `C_Login`, `C_Logout`, `C_OpenSession`, `C_CloseSession`, `C_CloseAllSessions`
- standard key/object lifecycle events such as `C_CreateObject`, `C_CopyObject`, `C_DestroyObject`, `C_GenerateKey`, `C_GenerateKeyPair`, `C_WrapKey`, `C_UnwrapKey`, `C_DeriveKey`
- standard crypto usage events such as `C_Encrypt*`, `C_Decrypt*`, `C_Digest*`, `C_Sign*`, `C_Verify*`
- HSM/audit-management events such as time/config/secret import-export actions
- Luna-only `CA_*` families such as cloning, HA, application/container access, external logging hooks, and other extension/admin workflows

That means Luna-native audit is a better source for questions like:

- "What did this HSM or partition see across all clients?"
- "Which Luna role performed the action?"
- "Did a Luna-only admin/control action happen outside our wrapper?"
- "Can we review a vendor-originated audit trail later?"

### What wrapper telemetry still does better

Wrapper telemetry remains better for:

- exact wrapper method/native call pairing from inside this process
- immediate in-process diagnostics and exception capture
- generic logging/tracing integrations (`ILogger`, `ActivitySource`, OpenTelemetry-style exporters)
- redaction rules controlled by this repo
- vendor-neutral observability that works across modules

So the right framing is:

- **wrapper telemetry** = application/runtime diagnostics
- **vendor-native audit** = HSM/platform accountability

## Thales Luna integration directions by path

The realistic ingestion paths are not equal. Some fit this repo much better than others.

| Path | What it means for Luna | Advantages | Main limitations | Recommendation |
| --- | --- | --- | --- | --- |
| CLI | Use LunaSH/Luna client commands to list/tail/archive/verify logs, likely from an external worker/script | Lowest conceptual barrier; does not require new native binding work | Appliance-specific, privilege-heavy, brittle to automate, poor fit for core wrapper library, not ideal for real-time ingestion | Acceptable for manual ops and possibly tooling prototypes; not the preferred first product integration |
| Remote syslog / remote audit stream | Configure Luna audit remote hosts and ingest the emitted stream in a collector | Most realistic near-real-time path for Luna Network HSM; centralizable; does not require changing PKCS#11 call flow; naturally separate from wrapper telemetry | Appliance-only path, parser/schema drift risk, operational setup required, transport/security/firewall concerns, log verification semantics still vendor-specific | **Best first direction for Luna Network HSM operational integration** |
| Export/archive | Use `audit log tarlogs`, transfer archives, optionally verify with audit secrets/HSM workflows | Preserves vendor file shape; good for compliance/offline review; verification story stays close to vendor model | Batch/offline, not real-time, secret/verification workflow is operationally heavy, awkward for generic library consumption | **Best conservative direction for offline import/review workflows** |
| API | Use a vendor-supported audit retrieval API | Strong fit when the vendor actually exposes one; structured responses can be easier to normalize | Product-specific: for Luna this appears realistic mainly for Luna Cloud HSM / DPoD, not the classic on-prem Network HSM audit path | **Good future path for Luna Cloud HSM / DPoD, but separate from on-prem Luna Network HSM work** |
| Direct PKCS#11 / `CA_*` binding for audit retrieval | Add Luna-specific code in or around the wrapper to pull audit data directly | Could theoretically reduce external dependencies if a stable read API existed | Public Luna docs reviewed here emphasize audit role commands, remote logging, export/verify, and cloud API variants rather than a simple generic PKCS#11 read path; mixing this into the core wrapper would blur boundaries | **Not recommended as the first implementation direction** |

## Detailed evaluation of each path

### 1. CLI-based collection

For Luna Network HSM, the public docs show that audit management is centered on LunaSH commands such as:

- audit configuration/login commands
- `audit log list`, `tail`, `tarlogs`, `verify`, `clear`
- `audit secret export` / import workflows
- remote-host configuration commands

This makes CLI automation a realistic **operational** path, but a poor **core-library** path.

Why it is not a good first-class wrapper feature:

- it depends on appliance roles, SSH/session context, and vendor command availability
- it is operationally sensitive and likely environment-specific
- it does not belong in `Pkcs11Wrapper` or `Pkcs11Wrapper.Native`, which are focused on PKCS#11 module usage rather than appliance shell automation
- it invites fragile parsing of command output instead of consuming a cleaner transport

Conservative conclusion:

- okay for manual runbooks, scripts, or proof-of-concept tooling
- not the preferred architectural center for future repo support

### 2. Remote syslog / remote audit stream

This is the strongest Luna Network HSM candidate for future integration.

Why it fits well:

- it leaves the core PKCS#11 wrapper unchanged
- it treats vendor audit as a separate ingestion stream, which matches the architectural distinction already described above
- it supports central collection and correlation with existing logging systems/SIEMs
- it can be near-real-time instead of batch-only

Why it still needs careful scoping:

- it is operationally configured on the Luna side, not discovered automatically by the wrapper
- message formats and facilities are vendor-specific and may vary by firmware/appliance generation
- transport choices (`udp`, `tcp`, `relp`, optional `tls`) affect deployment complexity and reliability
- correlation to wrapper telemetry will be approximate unless the vendor logs expose fields that map cleanly to local app context
- secure audit logging has vendor-managed semantics that should not be diluted into "just another app log"

Conservative conclusion:

- **best first direction for on-prem Luna audit ingestion**
- should live in an admin/infrastructure collector layer or separate tool, not in the base wrapper package

### 3. Export/archive + verification workflow

The Luna docs also make offline handling very explicit:

- rotate/archive logs
- transfer them off the appliance
- export/import the audit secret when needed
- verify logs on an HSM in the same domain / compatible verification setup

This is a strong fit for compliance and post-hoc review, but not for instant observability.

Why it matters for repo design:

- it suggests any future support for this path should preserve vendor provenance and verification metadata rather than flattening everything into normal app logs
- it implies that "audit import" and "audit verification" are separate concerns
- it argues for a future import/storage/reporting feature in the admin/infrastructure layer rather than in the PKCS#11 call wrappers

Conservative conclusion:

- strong candidate for future **offline import/review** work
- not a substitute for live telemetry

### 4. API-based retrieval

For classic Luna Network HSM, the public docs reviewed for this issue point much more strongly to:

- LunaSH audit commands
- remote logging
- archive/export/verification

For Luna Cloud HSM / DPoD, public material indicates an audit retrieval API that returns downloadable JSON/ZIP results for filtered audit queries.

That means "API integration" is real, but it should be treated as a **different product surface**.

Conservative conclusion:

- do not design a single Luna API integration as though it covered both on-prem Network HSM and cloud DPoD equally
- if future repo work targets Luna Cloud audit, scope that as a separate cloud-specific collector/integration issue

### 5. Direct vendor extension binding

Luna public docs show audit-related `CA_*` events and operations, including `CA_LogExternal` and audit-management families. However, that does **not** automatically mean the right next step is to add Luna extension binding just to claim "audit integration".

Reasons to avoid that as the first move:

- current repo architecture already keeps standard PKCS#11 separate from Luna-only `CA_*`
- audit retrieval/transport is still operationally centered on remote logging, archives, verification, and vendor-specific admin flows
- adding `CA_*` bindings would increase scope dramatically without solving the real ingestion/storage/correlation problem by itself
- `CA_LogExternal` is an audit-related capability, but it is not the same as a general, repo-ready audit collection strategy

Conservative conclusion:

- keep `CA_*` binding out of the first audit-integration wave
- only revisit it if a later issue proves a narrow, high-value Luna extension family is needed beyond transport/collection work

## Recommended repo direction

### 1. Keep wrapper telemetry and vendor-native audit as separate products inside the repo

Do **not** extend `IPkcs11OperationTelemetryListener` to pretend vendor-native audit records are the same event type.

Instead, future work should treat vendor audit as a separate pipeline with its own:

- source identification
- storage rules
- parser/collector implementation
- retention model
- correlation strategy
- trust/integrity notes

### 2. Keep vendor audit out of the `Pkcs11Wrapper` / `Pkcs11Wrapper.Native` core for now

The best fit for future implementation is the **admin/infrastructure side** or a separate optional tool/package, because vendor audit ingestion depends on operational transports and vendor-specific privileges rather than normal PKCS#11 call invocation.

That preserves the repo's current architecture:

- core wrapper = vendor-neutral PKCS#11 usage
- vendor-native audit = optional vendor-specific operational integration

### 3. Prefer a narrow common envelope over aggressive cross-vendor normalization

If future implementation work begins, it should preserve raw vendor details and normalize only a small shared envelope such as:

- vendor
- source family (syslog / archive / api)
- timestamp
- vendor action code
- status/outcome
- actor/role if present
- object/partition/device identifiers if present
- raw payload or vendor field map reference
- verification/provenance metadata

Do **not** try to invent a fake cross-vendor schema that erases Luna-specific meaning.

### 4. First future implementation slices should be transport-specific

The most realistic follow-up issues are:

1. **Luna Network HSM remote-audit ingestion evaluation spike**
   - define expected collector boundary
   - capture sample messages
   - design storage/correlation schema conservatively
2. **Luna archive import/review design**
   - define how archived audit files would be stored and surfaced
   - record verification status without forcing the repo to own all audit-secret workflows immediately
3. **Luna Cloud HSM / DPoD audit API evaluation**
   - explicitly separate cloud audit ingestion from on-prem appliance audit ingestion
4. **Admin-panel correlation UX design**
   - show wrapper telemetry, local admin audit, and vendor-native audit as separate but correlatable timelines rather than flattening them into one log type

## Out of scope for the first implementation wave

The following should remain out of scope unless a later issue narrows them explicitly:

- full vendor-audit ingestion in the core `Pkcs11Wrapper` package
- claiming that wrapper telemetry is compliance-grade HSM audit by itself
- automatic 1:1 correlation between wrapper telemetry events and vendor audit records
- direct Luna appliance/session automation from the core wrapper API
- cross-vendor "universal audit" semantics that hide vendor-specific meaning
- managing Luna audit secrets/import/export/verification end-to-end in an initial slice
- implementing broad Luna `CA_*` support just to say audit integration exists

## Practical bottom line

### What the repo already has

- strong wrapper-side PKCS#11 telemetry
- local admin-panel audit history with integrity checks
- conservative Luna compatibility/integration/extension-boundary docs

### What the repo does not yet have

- ingestion of vendor-native audit records
- a separate storage/correlation model for vendor-native audit
- Luna remote-audit or archive-import support
- a cloud-specific audit API collector for Luna Cloud HSM / DPoD

### Best next move after this issue

For **Luna Network HSM**, the most realistic first implementation direction is:

1. **remote audit/syslog ingestion** for near-real-time operational visibility
2. optionally later **archive import/review** for compliance/offline workflows

For **Luna Cloud HSM / DPoD**, the realistic direction is a **separate API-based collector path**, not the same design as on-prem appliance audit.

## Conclusion

Issue #41 should be treated as a **design/evaluation milestone**, not a hidden implementation issue.

The conservative architectural answer is:

- keep generic wrapper telemetry exactly what it already is: **redacted, vendor-neutral, process-local PKCS#11 observability**
- treat vendor-native audit as a **separate vendor-specific operational pipeline**
- for **Thales Luna Network HSM**, prioritize **remote audit/syslog** and later **archive import/review** over direct wrapper-core changes
- for **Luna Cloud HSM / DPoD**, evaluate a **separate API collector** rather than assuming the on-prem design applies unchanged
